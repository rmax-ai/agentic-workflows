## What changed

This iteration turned an implicit planning gap into explicit
repository memory. Until now, `.agent/backlog.yaml` and
`.agent/ontology-status.yaml` stopped at Phase 7 even though the
orchestrator prompt already spoke about later phases. That left
future work under-specified once the current matrix-refinement
stage eventually completes.

Phase 8 is now defined as the next breadth-expansion stage:
introduce additional canonical patterns when they represent truly
new reusable structure, add new modeled domains only when repeated
grounded evidence justifies them, and pair that broader surface
with linked grounded instances. A follow-on Phase 9 was then
defined as the place to deepen architecture, governance, and
consistency coverage after that breadth expansion stabilizes.

Because this is a durable sequencing rule rather than a temporary
plan note, it was recorded in
`.agent/decisions/D-0016-define-phase-8-as-breadth-expansion-after-coverage-refinement.md`.

## Reflection

The phase model is most useful when it distinguishes refinement of
the current ontology surface from deliberate broadening of that
surface. Naming Phase 8 explicitly around pattern, domain, and
instance expansion gives future iterations a clearer threshold for
when new structure is warranted and helps avoid treating every
remaining content opportunity as just another Phase 7 coverage fix.

No proposal was added because the durable rule belongs directly in
the decisions system and the phase-planning memory rather than in a
separate proposal artifact.

## Next step

definition to decide whether the next repository move should widen
the ontology with a new pattern-domain-instance cluster rather than
further rebalance the existing matrix.
